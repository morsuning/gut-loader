
## 文档约束

1. 任何项目都需要具备3个文档，统称为项目文档，分别是需求文档（PRD.md）、实现文档（IMPLEMENTATION.md）、README文档（README.md），每轮对话后都需要按需更新

- 如果新增了功能、细节导致了代码变更，问题修复除外，都应该将其说明更新至需求文档，位于项目根目录下的docs目录下，确保所有的功能特性都在需求文档中有描述，本文档需要作为行为驱动开放的参考文档，内容至少包含项目概述（简介、核心特性和解决的问题）、所有功能的说明和使用示例、技术要求和限制、使用注意事项等
- 如果实现了需求文档中的任何新增的功能、细节即进行了代码变更，都应该将其实现方案更新至实现文档，必要时需要采用UML图进行说明，有图就必须有详细说明，使用标准的mermaid格式，内容至少包括设计层面的技术架构（技术选型）、系统架构、模块详细说明、各类数据结构设计详细说明；功能层面的实现细节、难点解决方案、使用示例、配置示例、项目结构等，设计文档中不能出现实际代码，数据结构示例除外（如JSON、Protobuf等）。实现文档的内容必须与需求文档对应，如果不一致，则说明需求文档中有功能未实现
- README文档作为该项目的说明文档，遵循开源社区的README文档编写最佳实践，此外还需要包含该项目的介绍、亮点、使用说明、安装部署指南、配置指南、常见问题解答等内容，只在必要时更新
- 同样的，如果项目删去或调整了相关功能，则需要检查并更新项目文档，删去或调整该功能文档中的相应内容，要对原有相关内容进行原地修改，不用说修改、删除或新增了什么，或者提及在版本，只用维护文档的版本，直接说现在是什么样就行
- 需求文档和实现文档都应放入项目根目录下的docs目录下，README文档应放在项目根目录下
- 除README文档外，均禁止使用emoji，文档编写要遵循需求文档、实现文档的最佳实践且内容清晰、详细，覆盖项目的全部内容
- 单一文档如果超过2K行需要进行拆分
- 更新文档的任务总是在所有任务的最后完成

2. 每轮对话如果新增了功能代码或进行了问题修复等涉及代码的修改，都要添加进当前项目的变更记录，位置在项目根目录下的docs/changelog.md

- 项目变更记录中要有每次变更的版本号，采用语义化版本根据情况进行递增，当进行向下兼容的 bug 修复时修订号递增1，当添加向下兼容的新功能时次版本号递增1，当进行不兼容的 API 修改时主版本号递增1
- 如果是独立任务或者不涉及代码修改（只有注释修改也不算）则无需生成变更记录和更新项目文档
- 无需包含测试与验证内容
